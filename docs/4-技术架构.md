## 后端 Maven 多模块结构

```
ymq/
├── yudao-server/                ← 芋道启动模块（含 admin-api + app-api 双接口）
├── yudao-framework/             ← 芋道核心框架（自带 WxJava 集成）
├── yudao-module-system/         ← 芋道：用户/角色/字典/菜单（运营用）
├── yudao-module-infra/          ← 芋道：代码生成器、配置、监控
│
├── ymq-module-activity/         ← 业务：活动创建/接龙/对阵表（功能一·核心）
├── ymq-module-social/           ← 业务：发现附近/信用/黑名单/举报（功能一·扩展 + 全局基础设施）
├── ymq-module-match/            ← 业务：比赛记分（功能二）
├── ymq-module-rating/           ← 业务：段位评测 + AI 教练（功能三）
├── ymq-module-training/         ← 业务：训练成长（功能四）
├── ymq-module-content/          ← 业务：教学知识库内容（功能五，v1.2 新增）
└── ymq-module-poster/           ← 业务：海报生成（跨功能复用）
```

**架构原则**：所有业务都在芋道一个 Spring Boot 内运行，通过接口前缀路由：
- `/admin-api/system/**` → 芋道运营后台
- `/admin-api/ymq/**` → 我们的运营功能（活动审核、举报处理）
- `/app-api/ymq/**` → 小程序 API（C 端）

## 砍掉的芋道模块

芋道功能多，独立开发者要砍掉：
- `yudao-module-mall`（商城）
- `yudao-module-erp`（进销存）
- `yudao-module-crm`（客户管理）
- `yudao-module-bpm`（工作流）
- `yudao-module-ai`（用自己的 AI 教练实现替代）

→ 删掉后核心代码量降低 60%，编译速度提升明显。

## 部署架构

```
开发环境：本地 IDEA 启动 yudao-server（单进程）
测试环境：腾讯云轻量 2 核 4G（一台跑单进程）
生产环境（MVP）：腾讯云轻量 2 核 4G + 腾讯云 MySQL 1 核 1G
生产环境（v2，MAU > 1w）：
  - yudao-server 升级 4 核 8G
  - MySQL 升 2 核 4G
  - Redis 升 1G
生产环境（v3，MAU > 5w）：
  - 拆 yudao-server 为多实例 + 负载均衡
  - 拆出独立的 admin-server vs app-server
  - MySQL 主从、Redis 集群
```

## 已否决方案

| 方案 | 否决理由 |
|---|---|
| 一开始就上微服务（RuoYi-Cloud / yudao-cloud） | 独立开发者维护成本太高，MVP 不需要 |
| 一开始就上 K8s | 同上，腾讯云轻量足够 |
| 自训练羽毛球动作识别模型 | 数据/算力/时间都不允许，等 v3 用商业 API |
| macrozheng/mall 为骨架 | 已停更，且无小程序支持 |
| JeecgBoot 为骨架 | 低代码框架对手写场景反而是负担 |
| handsomestWei/ruoyi-vue-wxmini 直接 fork | 基于 2021 年版若依，太老 |
| Spring Cloud Gateway 做接口前缀路由 | 单进程 Spring Boot 已足够 |
| 双启动类（v1.1 方案） | 改用芋道后单启动 + 接口前缀更简单 |

## 决策变更流程

新增或修改技术决策：

1. 在本文件相应章节加新行（不要删除旧行，保留历史）
2. 在"决策时间"栏标注新版本号（如 v1.3）
3. 如影响 MVP 范围，同步更新 [2-MVP与验收.md](2-MVP与验收.md)
4. 如影响项目结构，同步更新 [README.md](../README.md)

**复议触发**：任何决策的"复议触发"条件满足时，必须在两周内 review，决定保持或变更，结论写入本文件。
