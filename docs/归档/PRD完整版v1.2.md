## 1. 产品定位

### 一句话介绍
**「让一群人一起打球这件事，从拉群到散场，全程顺滑。」**

### 详细描述
一款服务业余羽毛球社群的微信小程序，解决「组织一场球」全链路痛点：

- 召集人不用再手工排对阵表
- 队员能看清自己什么时候上场、和谁配
- **缺人时能让附近球友看到并申请加入**（v1.1 新增）
- 比赛能像专业赛事一样有比分页、有战报
- 打完球能看到自己的水平在涨
- **测完段位有个性化提升计划，4 周后复测看进步**（v1.2 新增）

### 与现有产品的差异化

| 现有 App | 主打 | 痛点 |
|---|---|---|
| 约球 App（约球、热门） | 陌生人约球（以人为锚） | 熟人圈不用，依赖城市规模，监管严 |
| 全民羽球 | 数据记录 | 重，单人打开 |
| 微信群 + 接龙 | 组织活动 | 只能接龙，不能排对阵 |
| **YMQ（我们）** | **熟人球队 + 高频组织工具 + 活动公开化** | **填补群和 App 之间的空白，"以活动为锚"辐射陌生球友** |

### 核心定位
> **不和「陌生人约球 App」竞争，做「微信群里的对阵表生成器 + 战报机器」，缺人时再把活动公开给附近球友。**
> 用户来自微信群，回到微信群——我们只是中间的工具层。

## 2. 北极星指标与阶段目标

### 北极星指标
**周活跃组织者数（WAO, Weekly Active Organizer）**——每周至少创建 1 场活动的用户数。
理由：召集人是这个产品的杠杆，1 个召集人能带来 6-12 个参与者。

### 阶段目标

| 阶段 | 时间 | 关键指标 |
|---|---|---|
| **MVP 验证** | 上线后 1 个月 | WAO ≥ 20，总用户 ≥ 300 |
| **种子增长** | 上线后 3 个月 | WAO ≥ 200，总用户 ≥ 5000，周留存 ≥ 35% |
| **冲月活破万** | 上线后 6 个月 | MAU ≥ 10000，WAO ≥ 800，自传播系数 K ≥ 1.2 |

### 辅助指标
- 接龙→对阵表生成转化率 ≥ 80%（这是核心价值传递）
- 分享卡片打开率 ≥ 25%（裂变效率）
- 段位评测完成率 ≥ 70%（爆款评测体验）
- **公开活动占比 ≥ 30%**（v1.1 新增，反映发现功能渗透率）
- **附近活动申请通过率 ≥ 50%**（v1.1 新增，反映匹配质量）

## 3. 用户画像

### 画像 A：球队队长 / 活动召集人（核心用户，KOC）

- **特征**：30–45 岁，男性偏多，单位 / 小区 / 培训班的"球队队长"
- **行为**：每周组织 1-3 场活动，建群、收钱、排场、记账
- **痛点**：
  - 接龙后人数不对要重排对阵
  - 每次新人加入要解释规则
  - 排对阵要照顾男女搭配、水平均衡
  - **临时缺人时只能在多个群里发广告求人**（v1.1 痛点）
- **核心诉求**：**省事**。一键搞定。
- **MAU 占比预估**：10%，但贡献 90% 的活动创建

### 画像 B：普通球友（基础用户，被动参与）

- **特征**：20–50 岁，男女均衡，每周打 1-2 次
- **行为**：在群里接龙、按队长安排上场、看比分
- **痛点**：
  - 不知道自己什么时候上场
  - 想知道今天和谁组队
  - 想看自己打了多少场
- **核心诉求**：**少操作**。点开就知道关键信息。
- **MAU 占比预估**：75%

### 画像 C：进阶玩家 / 比赛参与者

- **特征**：业余 5 级以上，热衷比赛
- **行为**：参加地区赛、单位赛、群里办的对抗赛
- **痛点**：
  - 比赛没有专业记分系统
  - 想看自己水平历史变化
  - 想知道自己段位
- **核心诉求**：**装备党 / 数据控**，喜欢数据和排名
- **MAU 占比预估**：15%

### 画像 D：散兵球友（v1.1 新增，发现模块的目标用户）

- **特征**：20–40 岁，刚搬到新城市 / 没有固定球队 / 朋友圈缺球友
- **行为**：搜索约球小程序、加入微信群、不固定打球
- **痛点**：
  - 想找附近的活动加入但不知道在哪
  - 加陌生人微信尴尬
  - 怕被骗、被拉黑、被骚扰
- **核心诉求**：**安全、可信地融入附近球友圈**
- **MAU 占比预估**：5%（早期），但增长潜力大

## 4. 竞品速览

| 产品 | 类型 | 核心功能 | 优势 | 劣势 | 我们的对策 |
|---|---|---|---|---|---|
| 约球 | App | 陌生人约球 | 城市渗透深 | 熟人圈不用、监管严、女性流失 | 不主打陌生人，做"以活动为锚的开放化"，更轻、更安全 |
| 全民羽球 | App | 个人数据 | 数据全 | 启动重、APP 门槛 | 用小程序无门槛 |
| 微信群 + 群接龙 | 小程序 | 接龙投票 | 入口自然 | 只能接龙 | 接龙后直接给对阵表 |
| 热门约球 | App | 场馆 + 教练 | 场馆资源 | 工具弱 | 工具优先，资源后补 |
| 球友汇 | 小程序 | 球队管理 | 已有用户 | 界面老旧 | 体验完胜 |

## 5. 核心用户旅程

### 旅程 1：召集人组织一场活动（最高频）

```
微信群里 → 我有场地，今晚 7-9 点
→ 召集人打开 YMQ 小程序 → 创建活动（填写：时间/场地/人数/模式 + 是否对外公开）
→ 生成接龙卡片，分享回群
→ 群友点链接 → 微信授权 → 自动加入活动
→ 召集人到点点"开始排对阵"
→ 系统按性别/水平/上场次数均衡 → 生成 X 轮对阵表
→ 召集人确认 → 一键转发"对阵表卡片"回群
→ 每个人能看到"自己 7:00 第 1 场 / 7:30 第 2 场 / 8:00 休息"
→ 打完点"散场" → 自动生成战报海报
```

### 旅程 2：参与者扫码进场

```
群里看到对阵表卡片 → 点进小程序 → 微信授权登录
→ 看到"今晚活动·我的对阵表"
→ 知道何时上场、和谁打、谁的对手
→ 打完场 → 在小程序里点比分 → 自动算积分变化
→ 看战报海报 → 转发朋友圈炫耀 → 拉新进来
```

### 旅程 3：段位评测（破圈裂变）

```
群里有人发"我测出来是 4.5 级" → 朋友点链接
→ 测评开始：10 道选择题 + 5 个动作选项
→ 5 分钟完成 → 给一个"测评中..."动画
→ 出报告卡片首页：「你是 4.3 级·进攻型」
→ "查看详细分析"按钮要求分享到朋友圈或群解锁
→ 解锁后看到 9 张精美图：水平定位/技术雷达图/进阶路线
```

### 旅程 4：办一场内部比赛

```
召集人 → 创建"比赛模式" → 选「接力赛 / 淘汰赛」
→ 接力赛：分两队、按顺序派人、累计 100 分
→ 淘汰赛：选 8/16/32 人、自动种子签 + 对阵图
→ 裁判端：手机扫码做裁判，点比分
→ 观众端：群里点链接看实时比分
→ 决出冠军 → 生成颁奖海报
```

### 旅程 5：散兵球友通过"发现"加入附近活动（v1.1 新增）

```
新球友打开小程序 → 底部 Tab "发现"
→ 授权位置 → 看到附近 3km 内"今晚还缺人"的公开活动
→ 点击"今晚 7 点·万体馆"卡片 → 查看详情
→ 点"申请加入" → 填段位 + 申请理由 → 提交
→ 召集人收到通知 → 看到申请人段位卡片 + 信用分 → 1 键通过
→ 球友收到"申请通过"通知 → 自动出现在对阵表
→ 当晚现场打球 → 第二天信用分 +1 → 形成正向循环
```

### 旅程 6：召集人临时缺人，开放活动求援（v1.1 新增）

```
召集人发现今晚少 1 人 → 进入活动详情
→ 点"对外公开"开关（限定半径 3km、限定段位 C1-B2、限定缺男）
→ 活动出现在附近球友的发现 Tab
→ 30 分钟内收到 3 个申请 → 看资料挑 1 个通过
→ 凑齐 6 人 → 关闭公开开关 → 继续按原流程进行
```

## 6. 模块一：活动接龙 + 智能对阵表

> **这是产品的核心，最大投入。**

### 6.1 角色

- **召集人**：发起活动、确认对阵、结束活动
- **参与者**：接龙加入、看自己的对阵
- **观众**：未登录用户可看活动公开页（吸引注册）

### 6.2 活动创建流程

#### 创建表单字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| 活动名称 | 文本 | 是 | 默认"YYYY-MM-DD 晚 7 点·XX 场馆" |
| 场馆名称 | 文本 | 是 | 可保存常用场馆 |
| 场馆地址 | 文本+地图选点 | 否 | 用于导航 + 附近发现 |
| 开始时间 | 时间 | 是 | |
| 结束时间 | 时间 | 是 | |
| 场地数量 | 数字 | 是 | 1-10 片 |
| 计划人数 | 数字 | 是 | 默认 6 人（1 片场地推荐） |
| 活动模式 | 单选 | 是 | 男双 / 女双 / 混双 / 自由组合 |
| 轮换规则 | 单选 | 是 | 固定搭档 / 每轮换搭档 |
| 单场用时 | 数字 | 是 | 默认 15 分钟 |
| AA 费用 | 数字 | 否 | 单人费用，用于二维码收款 |
| 活动备注 | 长文本 | 否 | 自由说明 |
| **对外公开** | 开关 | 否 | v1.1 新增，详见模块二 |

#### 业务规则

- 创建后生成唯一活动码（6 位短码）
- 默认有效期到结束时间 + 24 小时
- 同一召集人最多 5 个进行中活动
- 删除活动需要二次确认

### 6.3 接龙流程

#### 接龙卡片设计

```
┌──────────────────────────────────┐
│ 🏸 周三羽毛球·体育馆 3 号场       │
│                                  │
│ 📅 5/20（三）19:00-21:00         │
│ 📍 体育馆（导航）                 │
│ 👥 已报 4/6，还差 2 人            │
│ 💰 AA ¥25/人                     │
│ 🌐 公开活动（附近 3km 可见）     │
│                                  │
│ 已报名：                          │
│ 1. 张三 (男双·5 级)              │
│ 2. 李四 (女双·4 级)              │
│ 3. 王五 (男双·6 级)              │
│ 4. 赵六 (混双·5 级)              │
│                                  │
│ [我要报名]  [拉朋友来]           │
└──────────────────────────────────┘
```

#### 关键交互

- 参与者点击"我要报名"→ 微信授权 → 一秒加入
- 参与者可主动备注"我大概晚 30 分钟到"
- 召集人可设置候补名单（满员后排队）
- 召集人可移除某人（带原因，对方收到通知）
- 距离开始 2 小时未满员，召集人收到提醒"是否取消或继续 / **是否对外公开**"

#### 边界场景

- ✅ 临时加 1 个人：召集人可手动添加（无需对方登录，用昵称占位）
- ✅ 临时退出：参与者可在开始前 1 小时前自助退出
- ✅ 召集人临时改时间：所有参与者收到订阅消息通知
- ⚠️ 活动开始后退出：标记为"未到场"，影响信用分

### 6.4 对阵表生成（产品核心）

#### 触发时机

- 召集人在活动详情页点"生成对阵表"
- 系统校验：人数 ≥ 4，否则提示"至少 4 人才能开打"

#### 算法要点（详见 `algorithm.md`）

输入：
- 参与者列表（性别、水平、希望模式）
- 场地数 N
- 单场时长 T
- 总时长 H → 推算总轮数 R = H / T
- 模式（男双/女双/混双/自由）

输出（对阵表 JSON）：
```json
{
  "rounds": [
    {
      "roundNo": 1,
      "startTime": "19:00",
      "courts": [
        {
          "courtNo": 1,
          "team1": [{"userId": 1, "name": "张三"}, {"userId": 3, "name": "王五"}],
          "team2": [{"userId": 2, "name": "李四"}, {"userId": 4, "name": "赵六"}]
        }
      ],
      "resting": [{"userId": 5, "name": "周七"}, {"userId": 6, "name": "吴八"}]
    }
  ],
  "stats": {
    "playerStats": [
      {"userId": 1, "totalGames": 4, "restRounds": 2}
    ]
  }
}
```

#### 排程约束（按优先级）

1. **公平上场**：每人上场次数差 ≤ 1
2. **避免连场**：休息过的人优先上场，避免某人连打 3 场
3. **性别搭配**：混双模式严格 1 男 1 女配对
4. **水平均衡**：尽量让两边水平接近（v2 引入 ELO 后实现）
5. **多样搭配**：尽量不让同两个人连续两轮组队

#### 召集人可手动调整

- 拖动某人换位
- 锁定某场对阵（重新生成时不动）
- 一键"重新生成"按钮（保留锁定项）

### 6.5 对阵表分享卡片

转发到群的图片含：
- 活动名 / 时间 / 场馆
- 每轮对阵简表（场地 1 张三+王五 vs 李四+赵六 / 休息：周七）
- 二维码（扫码看自己的对阵）
- 小程序 logo + slogan

### 6.6 散场后流程

- 召集人点"结束活动"
- 自动算每人上场次数、参与时长
- 弹出"录入比分了吗？"→ 是→跳记分模块，否→直接生成战报
- 生成战报海报：
  - 今日 MVP（按胜场+分差）
  - 最佳搭档（胜场最多的组合）
  - 黑马（水平低但胜场高）
  - 上场最多的人

## 7. 模块二：发现附近开放活动

> **v1.1 新增模块。核心定位：不是"找人"，是"找球打"——让缺人的活动能被附近球友看到。**

### 7.1 设计目标

- 解决"晚上 7 点缺 1-2 个人凑不齐场"的真痛点
- 把熟人圈外延辐射到"小区 / 写字楼 / 3 公里内"的潜在球友
- **避免**变成约球软件的"陌生人社交"——以活动为锚，不暴露个人位置

### 7.2 与"约球类 App"的关键差异

| 维度 | 约球 App（陌生人）| 我们（活动公开化）|
|---|---|---|
| 主角 | 人 | 活动 |
| 入口 | 看附近的人 → 申请加好友 | 看附近缺人活动 → 申请加入 |
| 关系 | 私聊 → 慢慢熟 | 加入活动 → 现场认识 |
| 信任机制 | 个人头像 / 资料 | 召集人审核 + 信用记录 |
| 监管类目 | 社交·交友（审核严）| 体育·赛事（审核宽）|

### 7.3 公开活动开关（在活动创建表单中扩展）

| 字段 | 类型 | 默认 | 说明 |
|---|---|---|---|
| 是否对外公开 | 开关 | 关 | 开启后可被陌生球友发现 |
| 可见半径 | 单选 | 3km | 1km / 3km / 5km / 10km |
| 接受陌生人申请 | 开关 | 开 | 关闭后只是"被看到"，不接受申请 |
| 申请需审核 | 开关 | 开 | 关闭则陌生人可直接加入（不推荐）|
| 限制段位 | 多选 | 不限 | 例：仅 C1-B2 段位可申请 |
| 限制性别 | 单选 | 不限 | 缺男 / 缺女 |
| 限制人数 | 数字 | 1 | 通过陌生申请最多多少人 |

### 7.4 发现页（小程序底部 Tab "发现"）

#### 页面布局


#### 排序算法（详见 `algorithm.md`）

```
权重 = 距离权重 + 时间临近度 + 缺人紧迫度 + 召集人信誉
```

#### 过滤项

- 距离：1km / 3km / 5km / 10km
- 时间：今晚 / 明天 / 周末 / 全部
- 模式：男双 / 女双 / 混双 / 不限
- 缺人数：缺 1 / 缺 2+
- 段位：D / C / B / A / S

#### 默认行为

- 首次进入请求位置授权（软引导：用了能看附近）
- 拒绝授权 → 展示所在城市/区的热门公开活动
- 用户没设置城市 → 提示完善个人资料

### 7.5 申请加入流程

#### 球友视角

```
1. 发现页 → 点活动卡片 → 看完整详情
2. 看到「申请加入」按钮（已满则置灰显示"已满"）
3. 点击 → 弹出申请表单：
   - 姓名（默认用微信昵称）
   - 性别（必填）
   - 段位（必填，自动带入评测结果，可手动改）
   - 申请理由（选填，最多 50 字）
4. 提交 → 进入"待审核"状态，等召集人处理
5. 通过：收到订阅消息通知 + 自动加入活动 + 出现在对阵表
6. 拒绝：收到通知（可附原因）
```

#### 召集人视角

```
1. 活动详情页顶部出现"待审核申请 (X)"红点
2. 点击进入审核列表：
   ┌────────────────────────────────┐
   │ 王五 · 男 · C2 · 1.5km         │
   │ "我刚搬来附近，想找个固定球场" │
   │ 📊 信用：⭐⭐⭐⭐⭐ (5 场无放鸽子)│
   │ 🏆 历史：参加过 12 次活动      │
   │ [✓ 通过]  [✗ 拒绝]             │
   └────────────────────────────────┘
3. 1 键通过 → 自动加入活动名单
4. 拒绝 → 可选预设理由（人数已够 / 段位不符 / 其他）
```

### 7.6 信任与安全机制（关键）

| 机制 | 说明 |
|---|---|
| **实名要求** | 申请加入必须微信实名认证（小程序授权 phone）|
| **信用分** | 默认 100 分。放鸽子 -20，准时到场 +1，被召集人投诉 -30 |
| **黑名单** | 召集人可加 5 人黑名单（小范围使用）；信用分 < 60 进系统灰名单 |
| **首次惩罚轻** | 新用户首次放鸽子只警告不扣信用，避免误伤 |
| **举报机制** | 任何用户可举报骚扰/不当行为，运营 24h 内处理 |
| **女性保护** | 女性用户默认开启"仅显示已加入活动的男性头像"，可自定义 |

### 7.7 召集人激励

让召集人愿意"公开"活动：
- 公开活动比闭门活动多展示一个"⭐ 公开"标签，召集人形象更专业
- 召集人有"成长等级"（已办活动数、用户好评率），达到 LV5 解锁高级头衔
- 公开活动满员快 → 系统推送"附近还有 5 个人在看你的活动"，召唤感

### 7.8 边界场景

| 场景 | 处理 |
|---|---|
| 用户位置不准 | 允许手动选择常活动区域（保存为"我的球场"）|
| 公开活动满员 | 自动从"发现"页消失，已申请者收到"已满"通知 |
| 召集人临时取消 | 所有已通过申请的用户收到通知，并保留信用分 |
| 申请被拒后骚扰召集人 | 召集人可一键拉黑+举报 |
| 用户位置造假 | 检测异常移动速度，对疑似伪造的用户限流 |
| 同一活动同一申请人重复申请 | 限制 24 小时内一次 |

### 7.9 数据隐私

- 用户位置只存"最近一次进入发现页时的脱敏坐标"（精度到街道）
- 不持续追踪位置
- 位置数据不对外展示，只用于排序
- 用户可随时在设置里清除位置记录

- 两队（如 A 队 vs B 队），各派一定人数

- 累计到 100 分先到者胜
- 中途可以"接力换人"

```
4. 配置项目顺序（5 个项目可拖拽排序）
5. 配置每局分数（默认 21 分）
6. 配置总分上限（默认 100 分）
7. 开始比赛
```

#### 裁判端记分页

```
┌─────────────────────────────┐
│ 接力赛·公司年会             │
│ A 队 56 - 49 B 队           │
│                              │
│ 当前：男双 第 3 局           │
│ A 队 14 : 11 B 队            │
│                              │

│                              │
│ ⏸ 暂停    🔄 接力换人        │
│ 📋 历史项目                  │
└─────────────────────────────┘
```

- 大按钮加分（误触少）
- 长按减分（防误操作）
- 自动判断每局胜负、自动进入下一项
- 累计到 100 分自动锁分、宣布胜方

#### 观众端实时围观页

- 同样的比分大屏，无操作按钮
- WebSocket 实时同步
- 可分享"围观比分"链接给场外朋友

### 9.2 段位体系（自创清晰版）

| 周末未打卡 | 主动推送 | "本周还差 1 次打卡，加油！" |

### 10.7 周报与复测

#### 每周日推送周报

```
📊 本周训练报告

✅ 完成 3/3 次训练
🏆 累计 35 积分
🔥 连续打卡 14 天
📈 本周重点技能：杀球落点（已掌握 70%）

距离目标段位 C3 还差：
  • 反手挑球（练习 60% → 需要 80%）
  • 劈吊变化（练习 40% → 需要 70%）

[继续下周计划]
```

#### 计划结束触发复测

- 4 周计划结束后，自动弹出"复测段位"
- 复测后展示对比图：4 周前 C2·65 分 → 现在 C2·72 分 / C3·78 分
- 强烈鼓励分享对比海报到朋友圈

### 10.8 外链视频管理

- 每个外链视频存：原 URL + 平台 + 标题 + 时长 + 推荐理由
- 后台**每周自动检测**外链可用性（HTTP HEAD 请求）
- 失效自动替换备选 / 标记下架
- v1.5 自营内容上线后，外链作为补充

### 10.9 商业化路径（v1 全免费，先验证留存）

| 阶段 | 模式 | 价格 |
|---|---|---|
| MVP（v1） | 全免费 | ¥0 |
| v1.5 | 进阶会员（解锁 4 周以上长期计划 + 独家课）| ¥19-29/月 |
| v2 | 单课购买（教练独家）| ¥9.9-99/课 |
| v3 | 1v1 教练咨询 | 抽佣 15% |

### 10.10 v1 MVP 范围（Sprint 9）

- [ ] 技能树数据建模（4 大类，约 30 个技能）
- [ ] 训练计划生成算法（基于段位短板 + 简单规则）
- [ ] 4 周计划详情页 + 周/日任务展示
- [ ] 训练菜单文字 + 外链视频展示
- [ ] 打卡功能（每日 + 累计积分）
- [ ] 周报推送（微信订阅消息）
- [ ] 4 周结束触发复测段位
- [ ] 复测后对比海报

**v1 不做**：付费会员、独家视频、教练入驻、UGC、AI 个性化深度学习

### 10.11 边界场景

| 场景 | 处理 |
|---|---|
| 用户没做过段位评测 | 引导先去评测，否则计划生成给"通用版" |
| 用户改了目标段位 | 重新生成计划，原打卡记录保留 |
| 视频外链失效 | 显示"该视频暂不可用"+ 推荐备选 |
| 用户连续 14 天不打卡 | 推送召回 + 弹出"重新规划计划"选项 |
| 计划做完不复测 | 1 周后推送"复测看进步"+ 自动展示训练成果 |
| 同一技能反复练但不进步 | 显示"建议尝试不同动作或找教练" |

### 10.12 数据飞轮

从 v1 就要收集：
- 用户对每个训练菜单的"完成率"
- 用户对每个外链视频的"觉得有用"反馈
- 复测前后段位变化和具体训练的相关性

这些数据用于：
- v2 优化计划生成算法
- v3 招商教练入驻时的"哪种内容最有效"决策依据

## 12. 传播裂变机制设计

### 12.1 核心传播公式

```
新用户 = (现有用户 × 传播次数 × 转化率) - 流失
K 系数 ≥ 1.2 才能持续增长
```

### 12.2 每个核心动作都内嵌"分享钩子"

| 用户动作 | 触发的传播 | 设计 |
|---|---|---|
| 创建活动 | 接龙卡片到群 | 默认弹出"立即分享" |
| 生成对阵表 | 对阵表海报到群 | 美化海报，含二维码 |
| 比赛结束 | 战报海报 | 自动生成 MVP/最佳搭档 |
| 完成段位测试 | 段位卡片到朋友圈 | 分享解锁详细报告 |
| 看比赛 | 围观比分链接 | 看球的人邀请朋友一起看 |
| 加入新球队 | 球队战绩展示 | 球队主页可分享 |
| **公开活动满员** | "我办的球局又满了"动态 | v1.1 召集人荣誉感传播 |
| **散兵球友通过发现首次加入活动** | "我在 YMQ 上找到了第一场球" | v1.1 新用户故事性分享 |

### 12.3 关键裂变工具：海报模板系统

提前设计 30+ 套海报模板，覆盖：
- 5 套接龙卡片模板（节日主题、季节主题）
- 5 套对阵表模板
- 10 套战报模板（不同风格：极简/电竞/复古）
- 5 套段位评测报告模板
- 5 套比赛颁奖模板

让用户每次看到都觉得"诶这个挺好看"，提升转发欲。

### 12.4 排行榜玩法（v2 加）

- 小区周排行
- 球队周排行
- 全国段位排行
- 让用户为荣誉而打、为荣誉而分享

### 12.5 邀请激励（v2 加）

## 13. 非功能性需求

### 13.1 性能

- 小程序首屏 ≤ 1.5s（冷启动）
- 接龙报名响应 ≤ 300ms
- 对阵表生成 ≤ 2s（10 人以下）
- 实时比分同步延迟 ≤ 1s
- 发现页查询响应 ≤ 500ms（含地理位置查询）

### 13.2 可扩展性

- 数据库读写分离预留（v2）
- Redis 缓存热点数据（活动列表、排行榜、附近活动）
- 对阵表算法独立模块，方便后续替换
- 地理查询用 MySQL 8 空间索引（v2 切 PostGIS 或 Elasticsearch）

### 13.3 安全

- 微信登录 + JWT
- 所有 API 鉴权（除公开活动详情）
- 输入校验防 SQL 注入、XSS
- 敏感操作（删活动、移除人）需二次确认
- 防接口刷：限流 + 滑动验证（必要时）
- 位置数据加密存储

### 13.4 合规

- 用户协议 + 隐私协议（必须有，小程序审核要求）
- 收集信息最小化：只要 openid + 昵称 + 头像 + 性别（性别选填）
- **位置信息单独授权**，明确告知用途
- 不收集手机号（除非用户主动绑定）
- 不存储微信支付敏感信息
- ICP 备案
- 小程序类目：**体育→运动健身**（不申请"社交"类目）

### 13.5 监控

- 业务监控：活动创建数、用户增长、分享次数、附近活动申请通过率
- 技术监控：API 响应时间、错误率、服务器资源
- 错误追踪：Sentry

## 14. 技术架构选型

### 14.1 总体架构（v1.2 决策：切换到芋道）

> **基于 YunaiV/ruoyi-vue-pro（芋道源码，37.1k★）做骨架，单启动类 + 接口前缀区分运营后台和小程序 API。**

**v1.1 → v1.2 关键变化**：放弃 RuoYi-Vue-Plus，改用芋道——因为芋道原生包含微信小程序登录/支付/订阅消息/公众号全链路，对独立开发者节省 1-2 周自接 WxJava 的时间。同时放弃"双启动类"方案，采用芋道标准的"单启动 + 接口前缀"，更轻、更适合 MVP。

```

│             用户层                            │
├──────────────────────────────────────────────┤
│  微信小程序   |   H5 知识库   |   运营管理后台 │
│  (C 端用户)   |  (公开内容)   |  (运营/客服)   │
└────┬──────────────┬──────────────────┬───────┘
     │              │                  │
     │              ▼                  │
     │       ┌──────────┐              │

     │                                 │
     └────────────┬────────────────────┘
                  ▼
       ┌─────────────────────────────────┐
       │  yudao-server（单 Spring Boot） │
       │   ├─ /admin-api/* (运营后台)    │
       │   └─ /app-api/* (小程序)        │
       └─────────────┬───────────────────┘
                     │
       ┌─────────────┴───────────────────┐
       │  芋道核心 + 业务模块             │
       │  - yudao-module-system          │
       │  - yudao-module-infra           │
       │  - ymq-module-activity          │
```

### 14.2 后端 Maven 多模块结构

```
ymq/
├── yudao-server/                ← 芋道启动模块（含 admin-api + app-api 双接口）
├── yudao-framework/             ← 芋道核心框架（自带 WxJava 集成）
├── yudao-module-system/         ← 芋道：用户/角色/字典/菜单（运营用）
├── yudao-module-infra/          ← 芋道：代码生成器、配置、监控
│
├── ymq-module-activity/         ← 业务：活动 + 对阵表
├── ymq-module-match/            ← 业务：比赛记分
├── ymq-module-rating/           ← 业务：段位评测 + AI 教练
├── ymq-module-training/         ← 业务：训练成长 ⭐ v1.2 新增
├── ymq-module-social/           ← 业务：发现附近活动 + 信用 + 黑名单
└── ymq-module-poster/           ← 业务：海报生成
```

**架构原则**：所有业务都在芋道一个 Spring Boot 内运行，通过接口前缀路由：
- `/admin-api/system/**` → 芋道运营后台（账号/角色/字典等）
- `/admin-api/ymq/**` → 我们的运营功能（活动审核、举报处理）
- `/app-api/ymq/**` → 小程序 API（C 端）

### 14.5 砍掉的芋道模块（不用）

芋道功能多，独立开发者要砍掉：
- `yudao-module-mall`（商城）— 完全不需要
- `yudao-module-erp`（进销存）— 完全不需要
- `yudao-module-crm`（客户管理）— 完全不需要

### 14.6 部署架构

```

```

## 17. AI 能力路线

> AI 不是地基，是升级武器。地基是答题评测 + 海报裂变。AI 分四阶段引入，每阶段都要先验证用户买单再投入。

### 17.1 总体原则

| 原则 | 说明 |
|---|---|
| **分阶段** | 不一次性 all-in AI，按用户规模阶梯式释放 |
| **不影响段位** | AI 永远是辅助工具，不直接给段位评分（避免"测不准"投诉） |
| **裂变优先** | 每个 AI 功能都要内嵌分享钩子，做成"AI 看我打球说我哪里不对"的传播素材 |
| **成本可控** | 单次调用成本透明，配额机制防滥用 |
| **可回退** | 若 AI 服务故障，产品核心功能不受影响 |

### 17.2 四阶段路线图

| 阶段 | 时间点 | AI 能力 | 用户规模门槛 | 开发成本 | 月运营成本 |
|---|---|---|---|---|---|
| **v1** | Sprint 7-8（首发）| AI 教练问答彩蛋（图+文）| 0 | 3-5 天 | ¥500-2000 |
| **v1.5** | MAU 3000 后 | AI 教练强化（多轮对话、个人档案、推荐课程）| 3000 | 2-3 周 | ¥2000-8000 |
| **v2** | MAU 5w 后 | 单动作姿态分析（杀球/高远等独立动作 → 改进清单）| 50000 | 6-8 周 | ¥5000-2w |
| **v3** | MAU 20w 后 | 长视频段位评估（比赛录像 → 综合评分）| 200000 | 6-12 个月 | ¥3w-10w |

### 17.3 各阶段详细设计

#### v1：AI 教练问答（已在 §9.7 详述）

- 单图 + 单问，无多轮上下文
- 接 Claude / GPT-4o vision API
- 段位评测报告页彩蛋入口

#### v1.5：AI 教练强化

- **多轮对话**：用户可以连续追问"那我应该怎么练？"
- **个人档案接入**：AI 知道用户段位、近期战绩、上次咨询内容
- **推荐链接**：AI 回答末尾自动推荐知识库相关教程
- **教练形象**：给 AI 起名"小羽教练"，设计卡通形象
- **付费墙**：免费用户每月 5 次，付费会员无限（**第一个变现入口**）

#### v2：单动作姿态分析

- **核心交互**：用户拍 5-10 秒**单一动作**（杀球 / 高远球 / 网前小球）
- **技术栈**：MediaPipe BlazePose 提取人体 33 个关键点 → 时序数据 → 规则引擎对比理想动作
- **输出**：动作问题清单（不出段位评分）
  ```
  你的杀球动作分析（5 个问题，2 个亮点）：
  ⚠️ 引拍幅度偏小（理想 130°，你 95°）
  ⚠️ 击球点偏低（理想头顶斜前方，你头顶正上方）
  ✅ 转体到位
  💡 改进建议视频：[杀球引拍标准动作 2 分钟]
  ```
- **关键工程难题**：
  - 用户拍摄角度不一（需要引导 + 自动判别"侧面/正面/俯视"）
  - 不同身高的人骨架差异（用归一化坐标）
  - 球拍/羽毛球本身不被识别（用动作而非器械判断）

#### v3：长视频段位评估

- **前提**：要么自训练模型（积累 5w+ 标注样本），要么接腾讯云/阿里云成熟的体育视频分析 API（届时商业方案应该有了）
- **形态**：用户上传 2-5 分钟比赛视频 → AI 综合评分 → 给"参考段位"
- **关键设计**：永远叫"参考段位"，不替代答题段位

### 17.4 v1 AI 教练彩蛋的技术细节

#### API 选型

| 候选 | 优点 | 缺点 | 推荐度 |
|---|---|---|---|
| **Claude 4.6/4.7 (multimodal)** | 中文好、回答结构化、安全护栏强 | 国内访问需中转 | ⭐⭐⭐⭐ |
| GPT-4o | 同上，国内中转 | 中文略弱于 Claude | ⭐⭐⭐⭐ |
| 国内多模态（豆包/Qwen-VL）| 国内直连快 | 体育领域理解略弱 | ⭐⭐⭐ |

**推荐**：v1 用 Claude API（多模态），通过国内 API 中转。后续按成本切换。

#### Prompt 模板（关键）

```
你是"小羽教练"，一位耐心的业余羽毛球教练。
用户上传了一张关于「{type}」的图片，可能还附文字描述。

请按以下格式输出：

1. ✅ 做得好的（1-2 点，鼓励为主）
2. ⚠️ 需改进的（1-3 点，具体可执行）
3. 💡 改进建议（1-2 句话）
4. 📺 推荐练习方向（简短，不超过 20 字）

要求：
- 称呼"球友"，不要说"用户"
- 语气友好不批判
- 如果图片看不清，礼貌请求重拍（不强行点评）
- 不评段位、不和其他选手对比
- 不超过 200 字
- 结尾不要任何免责声明（系统会自动加）

用户描述：{user_text}
图片类型：{type}  // 握拍 / 站位 / 装备 / 其他
```

#### 数据存储（最小）

```sql
-- 简单记录，不存图片内容
CREATE TABLE `t_ai_coach_log` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `user_id` BIGINT UNSIGNED NOT NULL,
  `question_type` VARCHAR(16) NOT NULL,  -- GRIP/STANCE/EQUIPMENT/FREE
  `user_text` VARCHAR(500),
  `image_url` VARCHAR(512),  -- COS 中存 7 天后清理
  `ai_response` TEXT NOT NULL,
  `cost_cents` INT DEFAULT 0,  -- 此次调用成本，分
  `user_feedback` TINYINT DEFAULT NULL,  -- 1有用 0没用
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_user_time` (`user_id`, `created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='AI 教练对话记录';
```

#### 成本预估（v1）

- 假设每次调用 ¥0.05-0.15（Claude Sonnet vision）
- 每日 500 次调用 → ¥25-75/日 → 月 ¥750-2250
- 月活 1 万时按月预算 ¥2000 控制

### 17.5 关键风险

| 风险 | 应对 |
|---|---|
| AI 回答不准/胡说 | Prompt 严格约束，加"看不清请重拍"安全话术，用户反馈循环优化 |
| 用户用 AI 测段位（绕过答题）| 严格分离：AI 教练页永远不出现"你的段位是 X"的文字 |
| API 成本失控 | 配额机制 + 监控告警，超出预算自动降级到固定话术 |
| 图片含敏感内容 | 微信内容安全 API 前置过滤 + 自建敏感词库 |
| 国际 API 中断 | 准备国内 API 备用（豆包/Qwen-VL）一键切换 |

### 17.6 数据飞轮（v2 准备）

从 v1 开始就要收集：
- 用户上传的图片（脱敏后）
- AI 回答 + 用户反馈"有用/没用"
- 同段位用户的常见问题聚类

这些数据在 v2 阶段是训练自己模型的种子数据，**现在不收集就晚了**。

## 附：术语表

- **WAO**：周活跃组织者数
- **K 系数**：每个用户带来的新用户数
- **接龙**：微信群里发起的报名方式

- **技能树**：本产品对羽毛球技能的分类体系（4 大类约 30 个技能），训练计划基于此生成
- **WxJava**：binarywang/WxJava，Java 处理微信登录/支付/订阅消息的事实标准 SDK
