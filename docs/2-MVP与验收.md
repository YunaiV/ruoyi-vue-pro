## MVP 定位与边界

### 一句话定位
> 召集人创建活动后，群友可以低成本报名，系统能生成一张**比手工排场更省事、更公平、更容易分享**的对阵表。

### MVP 不是
- 完整羽毛球生态
- 装备社区
- 陌生人约球
- 训练营

### MVP 是
- 一个能让微信群组织者立刻省事的工具
- 一张能在群里被自然转发的对阵表

### 北极星指标
仍是 WAO（周活跃组织者数）。MVP 验证目标：

| 指标 | 目标 | 说明 |
|---|---:|---|
| 种子球队 | 10 个 | 真实微信群使用 |
| WAO | 20 | 上线后 1 个月 |
| 总用户 | 300 | 主要来自接龙和对阵表分享 |
| 接龙→对阵表生成转化率 | 80% | 证明核心价值触达 |
| 分享卡片打开率 | 25% | 证明微信群传播链路有效 |

### 服务的角色

| 优先级 | 角色 | MVP 必须支持的动作 |
|---|---|---|
| P0 | 球队队长 / 召集人 | 发起活动、让群友报名、生成对阵表、转发回群、临时调整人员 |
| P1 | 普通球友 | 点开群分享卡片、微信授权登录、报名、查看自己的对阵 |
| P2 | 观众 / 未登录用户 | 可看公开活动详情，但不专门建设 |

## MVP 做什么

| 优先级 | 功能 | 范围 |
|---|---|---|
| P0 | 微信登录 | 小程序授权登录，生成 C 端用户身份 |
| P0 | 创建活动 | 时间、场馆、人数、场地数、模式、单场时长、备注 |
| P0 | 活动短码 | 创建后生成唯一 6 位短码 |
| P0 | 接龙报名 | 从分享卡片进入可一键报名 |
| P0 | 候补/满员 | 满员进入候补或提示，召集人可管理 |
| P0 | 手动占位 | 召集人可添加未登录球友昵称占位 |
| P0 | 退出/移除 | 开始前允许自助退出，召集人可移除并填原因 |
| P0 | 对阵表生成 | 双打，优先公平上场、休息均衡、基础性别约束 |
| P0 | 我的对阵 | 普通球友看到自己每轮上场、搭档、对手、休息 |
| P0 | 手动调整 | 召集人可拖动换位或重新生成 |
| P0 | 分享对阵表 | 适合微信群传播的卡片或海报 |
| P1 | 结束活动 | 召集人结束活动并生成基础战报 |
| P1 | 基础后台 | 运营后台可查看活动、用户、报名记录 |

## MVP 不做什么

| 暂不做 | 原因 | 回归时机 |
|---|---|---|
| 附近发现完整版 | 依赖用户密度，早做空列表 | 种子球队稳定后灰度 |
| 活动公开申请审核完整版 | 不是第一版核心 | Sprint 4 |
| 接力赛 / 淘汰赛记分 | 使用频率未知 | 核心组织链路跑通后 |
| 段位评测 | 有裂变潜力但非主链路 | 需增长实验时 |
| AI 教练 | 趣味和传播大但不应影响地基 | 段位评测上线后作为彩蛋 |
| 训练成长 | 留存模块，依赖段位数据 | 评测有足够样本后 |
| AA 收款 | 涉及支付、退款、合规 | 运营验证强需求后 |
| 复杂球队体系 | 抬高建模成本 | 有 10 个以上活跃球队后 |
| ELO 战绩积分 | 需比赛数据积累 | 比赛记分上线后 |

## 第一版核心用户旅程

```
召集人在微信群说今晚打球
→ 打开 YMQ 创建活动
→ 分享报名卡片到微信群
→ 群友点卡片登录并报名
→ 到点后召集人点击生成对阵表
→ 系统生成多轮对阵和休息安排
→ 召集人调整后分享对阵表回群
→ 球友点开只看自己的安排
→ 散场后召集人结束活动并生成基础战报
```

---

## 关键产品约束

- 普通球友路径要极短，不要求先完善大量资料
- 召集人必须能手动兜底，算法不完美时也不失信任
- 分享卡片必须能独立传递价值，不打开小程序也能大致看懂
- MVP 默认服务熟人微信群，不主动强化陌生人社交
- 对阵算法第一版求**稳定、可解释、可调整**，不追求最优解

## 退出 MVP 的门槛

满足以下条件后，再进入发现、比赛、段位等增长模块：

| 条件 | 判断方式 |
|---|---|
| 10 个种子球队完成真实活动 | 每个球队至少组织 2 场 |
| 召集人愿意复用 | 50% 以上召集人在 2 周内创建第二场 |
| 对阵表投诉可控 | 每 10 场活动中，因对阵不公平导致重排的少于 2 场 |
| 分享链路有效 | 报名用户 70% 以上来自微信群分享入口 |
| 核心链路稳定 | 创建、报名、生成、分享、结束无 P0 阻断问题 |

## 验收标准

### 验收口径

- **P0**：不满足则 MVP 不能上线
- **P1**：不满足可灰度，但必须记录风险
- **P2**：锦上添花，不阻塞上线

### 1. 用户与登录

| 编号 | 优先级 | 验收标准 |
|---|---|---|
| U-01 | P0 | 用户首次进入小程序时，可以通过微信授权完成登录 |
| U-02 | P0 | 同一微信用户重复进入时，不会创建重复 C 端用户 |
| U-03 | P0 | 普通球友报名时，昵称和头像可从微信信息带入 |
| U-04 | P1 | 用户可补充性别和段位；未补充时仍可报名自由组合活动 |
| U-05 | P0 | C 端 `t_user` 与芋道 `system_users` 完全隔离；C 端登录返回的 token 不能访问 `/admin-api/*`，反之亦然 |

### 2. 活动创建

| 编号 | 优先级 | 验收标准 |
|---|---|---|
| A-01 | P0 | 召集人可创建活动，必填项包括活动名称、场馆、开始时间、结束时间、场地数、计划人数、活动模式、单场时长 |
| A-02 | P0 | 创建成功后系统生成唯一 6 位活动短码 |
| A-03 | P0 | 活动结束时间必须晚于开始时间，否则阻止提交并提示 |
| A-04 | P0 | 场地数限制 1-10 片，计划人数 ≥ 4 |
| A-05 | P0 | 同一召集人最多存在 5 个进行中活动 |
| A-06 | P1 | 召集人可保存常用场馆并在下次创建时复用 |
| A-07 | P1 | 召集人可修改未开始活动的时间、场馆、人数、备注 |
| A-08 | P1 | 修改关键字段后，已报名用户能看到最新活动信息 |
| A-09 | P0 | 后端启动后，`/admin-api/*` 和 `/app-api/*` 均可访问，且分别走独立的 SecurityConfig；未授权请求返回 401 |

### 3. 接龙报名

| 编号 | 优先级 | 验收标准 |
|---|---|---|
| J-01 | P0 | 召集人可把活动报名卡片分享到微信群 |
| J-02 | P0 | 球友从分享卡片进入后，可完成登录并报名 |
| J-03 | P0 | 同一用户不能重复报名同一活动 |
| J-04 | P0 | 活动详情展示当前报名人数、计划人数、已报名名单、活动状态 |
| J-05 | P0 | 满员后继续报名时，用户进入候补或收满员提示 |
| J-06 | P0 | 召集人可手动添加未登录球友作为占位成员 |
| J-07 | P0 | 召集人可移除成员，移除时需填写或选择原因 |
| J-08 | P1 | 参与者可在活动开始前 1 小时自助退出 |
| J-09 | P1 | 活动开始后退出会标记异常状态，后续可接信用分 |

### 4. 对阵表生成

| 编号 | 优先级 | 验收标准 |
|---|---|---|
| M-01 | P0 | 人数 < 4 时，系统不允许生成对阵表，提示至少 4 人 |
| M-02 | P0 | 系统根据活动总时长和单场时长计算轮数 |
| M-03 | P0 | 每场双打必须由 4 人组成，同一轮同一人最多上场 1 次 |
| M-04 | P0 | 每位成员总上场次数差 ≤ 1，除非人数和场地组合天然无法满足 |
| M-05 | P0 | 休息过的人在下一轮有更高优先级上场 |
| M-06 | P0 | 混双模式下，每队优先满足 1 男 1 女；人数不满足时给出明确提示 |
| M-07 | P0 | 6 人 1 片场地 8 轮可以生成对阵表，且每轮 4 人上场、2 人休息 |
| M-08 | P0 | 8 人 2 片场地 6 轮可以生成对阵表，且每轮全员上场 |
| M-09 | P0 | 4 人 1 片场地时，每轮全员上场，无休息列表 |
| M-10 | P1 | 系统尽量避免同一用户连续上场超过 2 轮 |
| M-11 | P1 | 系统尽量避免同两人连续两轮搭档 |
| M-12 | P1 | 生成耗时：10 人以下 ≤ 500ms，20 人以下 ≤ 2s |

### 5. 对阵表调整与查看

| 编号 | 优先级 | 验收标准 |
|---|---|---|
| E-01 | P0 | 召集人可查看完整轮次、场地、队伍和休息名单 |
| E-02 | P0 | 普通球友可只看"我的对阵"，包括上场时间、搭档、对手、休息轮次 |
| E-03 | P0 | 召集人可手动交换两个位置上的成员 |
| E-04 | P1 | 召集人可锁定某场对阵，重新生成时保留锁定内容 |
| E-05 | P1 | 手动调整后系统能重新计算每人上场次数统计 |

### 6. 分享与战报

| 编号 | 优先级 | 验收标准 |
|---|---|---|
| S-01 | P0 | 活动报名卡片包含活动名、时间、场馆、已报人数、报名入口 |
| S-02 | P0 | 对阵表分享卡片包含活动名、时间、场馆、轮次简表、进入入口 |
| S-03 | P0 | 分享卡片被打开后，用户能进入对应活动，不会丢失上下文 |
| S-04 | P1 | 召集人结束活动后，系统生成基础战报，包括上场次数、参与人数、活动时间 |
| S-05 | P2 | 战报包含 MVP、最佳搭档、黑马等趣味标签 |

### 7. 管理后台

| 编号 | 优先级 | 验收标准 |
|---|---|---|
| O-01 | P0 | 运营后台可查看活动列表、活动详情、报名成员 |
| O-02 | P0 | 运营后台可查看 C 端用户列表 |
| O-03 | P1 | 运营后台可按活动状态、创建时间、召集人筛选活动 |
| O-04 | P1 | 运营后台可下线异常活动 |
| O-05 | P1 | 运营后台 `system_users` 默认账号（admin/admin123 等）上线前必须改密码并删除示例账号 |

### 8. 非功能验收

| 编号 | 优先级 | 验收标准 |
|---|---|---|
| N-01 | P0 | 小程序核心页面冷启动首屏目标 ≤ 1.5s |
| N-02 | P0 | 接龙报名接口目标响应 ≤ 300ms |
| N-03 | P0 | 所有非公开 API 必须鉴权 |
| N-04 | P0 | 删除活动、移除成员等敏感操作需二次确认 |
| N-05 | P0 | 位置授权、手机号绑定、支付等非 MVP 能力不能阻塞核心链路 |
| N-06 | P1 | 接入错误追踪，至少能定位 API 错误和小程序前端异常 |
| N-07 | P0 | WxJava 配置（appId/secret/mchId/apiKey 等）必须从环境变量或外部配置中心读取，不得提交到 git |
| N-08 | P1 | 砍掉芋道 mall/erp/crm/bpm/ai 模块后，工程能成功编译和启动，不报缺失依赖 |

## 演示验收脚本

上线前至少跑通以下 3 条演示脚本：

### 脚本一：6 人 1 场地活动

```
创建活动：今晚 19:00-21:00，1 片场地，6 人，15 分钟一场
→ 6 名用户报名
→ 生成 8 轮对阵
→ 验证每轮 4 人上场、2 人休息
→ 验证每人上场次数差不超过 1
→ 分享对阵表
→ 普通用户打开"我的对阵"
→ 结束活动生成基础战报
```

### 脚本二：8 人 2 场地混双活动

```
创建活动：2 片场地，8 人，混双模式
→ 4 男 4 女报名
→ 生成对阵表
→ 验证每场每队为 1 男 1 女
→ 手动交换两个成员
→ 验证统计更新
```

### 脚本三：边界异常

```
创建活动并只报名 3 人
→ 点击生成对阵表
→ 系统阻止并提示至少 4 人
→ 添加 1 个手动占位成员
→ 再次生成成功
→ 移除成员时出现二次确认和原因选择
```
