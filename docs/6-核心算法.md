## 1. 对阵表生成算法（产品核心）

### 1.1 问题定义

**输入**：
- `members`: 参与者列表 `[{id, name, gender, level}]`
- `courtCount`: 场地数 N
- `matchDuration`: 单场时长（分钟）
- `totalMinutes`: 活动总时长
- `mode`: `MEN_DOUBLES` / `WOMEN_DOUBLES` / `MIXED` / `FREE`
- `rotation`: `FIXED`（固定搭档）/ `ROTATE`（每轮换）

**输出**：
- `rounds`: 每一轮的场地分配 + 休息名单
- `stats`: 每个人的上场统计

### 1.2 约束（按优先级）

| 优先级 | 约束 | 必须 / 软 |
|---|---|---|
| P0 | 每场 4 人（双打）| 必须 |
| P0 | 同一轮内每人最多上一场 | 必须 |
| P1 | 混双模式：每队严格 1 男 1 女 | 必须 |
| P2 | 每人总上场次数差 ≤ 1 | 必须 |
| P3 | 避免连续上场超过 2 场 | 软 |
| P4 | 同两人不连续两轮组队 | 软 |
| P5 | 两队水平接近（v2，引入 ELO 后）| 软 |

### 1.3 算法选型

**方案：贪心 + 局部回溯**

> 不用 ILP 求解器（依赖重），不用 SAT（学习成本高）。
> 业务规模小（一般 6-20 人、3-8 轮），贪心 + 启发式足够。

### 1.4 核心伪代码

```java
public MatchupPlan generate(MatchupRequest req) {
    int rounds = req.totalMinutes / req.matchDuration;
    int playersPerRound = req.courtCount * 4;
    List<Member> members = req.members;

    // 初始化每个人的"上场计数器"和"上次状态"
    Map<Long, PlayerStats> stats = initStats(members);
    List<Round> result = new ArrayList<>();
    Set<Pair> recentPartnerships = new HashSet<>();

    for (int r = 1; r <= rounds; r++) {
        Round round = scheduleOneRound(members, stats, recentPartnerships, req);
        result.add(round);
        updateStats(round, stats);
        updateRecentPartnerships(round, recentPartnerships, r);
    }

    return new MatchupPlan(result, stats);
}

private Round scheduleOneRound(...) {
    // 1. 按 (上场次数 ASC, 连续休息轮数 DESC) 排序候选
    List<Member> candidates = sortByPriority(members, stats);

    // 2. 选出本轮要上场的 playersPerRound 个人
    List<Member> playing = pickTopN(candidates, playersPerRound);
    List<Member> resting = candidates.subList(playersPerRound, candidates.size());

    // 3. 根据 mode 分组到各场地
    List<Court> courts = assignToCourts(playing, req.mode, recentPartnerships);

    // 4. 若违反混双性别约束，触发局部回溯：换一个人重试
    if (!validateGenderConstraint(courts, req.mode)) {
        return retryWithBacktrack(...);
    }

    return new Round(r, courts, resting);
}
```

### 1.5 关键细节

#### 1.5.1 候选排序权重

```java
double priority(Member m, PlayerStats s, int currentRound) {
    // 上场次数越少优先级越高
    double playWeight = -s.totalGames * 10.0;
    // 连续休息轮数越多优先级越高
    double restWeight = s.consecutiveRestRounds * 5.0;
    // 避免连续 3 场以上：若已连续 2 场，强烈降低优先级
    double burnoutPenalty = s.consecutivePlayRounds >= 2 ? -20.0 : 0;
    return playWeight + restWeight + burnoutPenalty;
}
```

#### 1.5.2 混双性别约束处理

```java
// 选出 playersPerRound 人后，按性别分组
List<Member> males = filter(playing, m -> m.gender == MALE);
List<Member> females = filter(playing, m -> m.gender == FEMALE);

// 混双需要 男:女 = N:N (每场 1 男 1 女)
int neededPairs = courtCount * 2;
if (males.size() < neededPairs || females.size() < neededPairs) {
    // 不够 → 从候选区找一个匹配性别的人换进来
    Member swapIn = findSwap(candidates, /*需要的性别*/);
    swap(playing, /*移除*/, swapIn);
}

// 分配到场地：每场 (男1, 男2) vs (女1, 女2)? 不对，混双是 (男1+女1) vs (男2+女2)
for (int c = 0; c < courtCount; c++) {
    Court court = new Court(c);
    court.team1 = [males.get(c*2), females.get(c*2)];
    court.team2 = [males.get(c*2+1), females.get(c*2+1)];
}
```

#### 1.5.3 避免重复搭档

```java
// 在分配到场地时，遍历可能的配对，跳过最近 2 轮内组队过的
boolean isRecentlyPartnered(Member a, Member b, Set<Pair> recentPartnerships) {
    return recentPartnerships.contains(new Pair(a.id, b.id));
}
```

### 1.6 边界场景处理

| 场景 | 处理 |
|---|---|
| 人数 < 4 | 拒绝生成，提示"至少 4 人" |
| 人数 = 4 | 每轮所有人都上，无休息 |
| 混双模式但只有 1 个女生 | 提示"混双模式至少需要 2 男 2 女" |
| 总轮数过多导致每人 > 20 场 | 提示"建议缩短时长或加场地" |
| 人数恰好等于 4 × 场地数 | 无休息，全员每轮上场 |

### 1.7 性能要求

- 10 人以下 5 轮：≤ 100ms
- 20 人 8 轮：≤ 500ms
- 30 人 10 轮：≤ 2s

## 2. 接力赛积分算法

### 2.1 规则

- 两队对战
- 按预设顺序进行 N 个项目（默认 5 项：男单/女单/男双/女双/混双）
- 每个项目打到 X 分制（默认 21 分）
- 项目结束后，将分差累计到队伍总分
- 累计到 Y 分先到者胜（默认 100 分）

### 2.2 计算逻辑

```java
public class RelayScoreEngine {
    public RelayState scoreUpdate(RelayState state, int teamIndex, int delta) {
        // 1. 单局分加 1
        state.currentGameScore[teamIndex] += delta;
        int[] scores = state.currentGameScore;

        // 2. 检查单局是否结束（21 分，差 2 分）
        if (isGameOver(scores, state.scorePerGame)) {
            int winner = scores[0] > scores[1] ? 0 : 1;
            int diff = Math.abs(scores[0] - scores[1]);

            // 3. 累计分差到队伍总分
            state.teamTotalScores[winner] += diff;

            // 4. 检查总分是否触顶
            if (state.teamTotalScores[winner] >= state.totalScoreCap) {
                state.winner = winner;
                state.status = FINISHED;
                return state;
            }

            // 5. 进入下一项目
            state.currentItemIndex++;
            if (state.currentItemIndex >= state.items.size()) {
                state.currentItemIndex = 0;  // 循环
            }
            state.currentGameScore = new int[]{0, 0};
        }
        return state;
    }

    private boolean isGameOver(int[] s, int target) {
        // 21 分制：先到 21 且领先 2 分，封顶 30
        int max = Math.max(s[0], s[1]);
        int min = Math.min(s[0], s[1]);
        return (max >= target && max - min >= 2) || max >= 30;
    }
}
```

### 2.3 关键场景

- 暂停 / 继续：state.status = PAUSED，记分按钮置灰
- 接力换人：替换 state.items[currentItemIndex].players，不影响当前比分
- 撤销：维护一个 history stack，最多撤销 10 步

## 3. 淘汰赛对阵生成

### 3.1 种子签算法

#### SNAKE 蛇形（默认推荐）

```
1. 按水平降序排列选手 [p1, p2, p3, ..., pN]
2. 分配到 N/2 个半区位置：
   位置 1: p1     位置 5: p5
   位置 2: p4     位置 6: p8
   位置 3: p5     位置 7: p9
   位置 4: p8     位置 8: p12
   ...
3. 确保强队分散到不同 1/4 区
```

#### RANDOM 完全随机

```java
Collections.shuffle(participants);
```

#### MANUAL 手动

由召集人拖动安排。

### 3.2 对阵图生成

```java
public Bracket generateBracket(int participantCount, List<Player> seededPlayers) {
    int rounds = (int) (Math.log(participantCount) / Math.log(2));  // 8 人 → 3 轮
    Bracket bracket = new Bracket();

    // 第一轮
    Round round1 = new Round(1);
    for (int i = 0; i < participantCount; i += 2) {
        round1.matches.add(new Match(seededPlayers.get(i), seededPlayers.get(i + 1)));
    }
    bracket.rounds.add(round1);

    // 后续轮次（先建空架子，比赛结束时回填胜者）
    int matchesInRound = participantCount / 4;
    for (int r = 2; r <= rounds; r++) {
        Round round = new Round(r);
        for (int i = 0; i < matchesInRound; i++) {
            round.matches.add(new Match(null, null));
        }
        bracket.rounds.add(round);
        matchesInRound /= 2;
    }

    return bracket;
}
```

### 3.3 胜者推进

```java
public void advanceWinner(Bracket bracket, int round, int matchIndex, Player winner) {
    Match nextMatch = bracket.rounds.get(round).matches.get(matchIndex / 2);
    if (matchIndex % 2 == 0) {
        nextMatch.player1 = winner;
    } else {
        nextMatch.player2 = winner;
    }
}
```

### 3.4 非 2^N 参赛人数处理

例：12 人参赛
- 上升到 16 人空位（4 个轮空）
- 轮空位置给种子排名前 4 的选手
- 第一轮：12 → 8（轮空的 4 人直接进 8 强）

## 4. 段位评测算法

### 4.1 评分公式

```
总分 = 基础分 + 技术分 + 战术分 + 经验分

基础分 (0-10) = 球龄分(0-5) + 频率分(0-5)
技术分 (0-50) = Σ(技术题得分 × 题目权重) × 归一化系数
战术分 (0-20) = Σ(战术题得分 × 题目权重) × 归一化系数
经验分 (0-10) = max(单项最高得分, 经验值平均) × 系数

最终段位 = mapScoreToLevel(总分)
```

### 4.2 段位映射

```java
private static final TreeMap<Integer, String> LEVEL_MAP = new TreeMap<>() {{
    put(0, "D1");    // 0-15
    put(16, "D2");   // 16-25
    put(26, "D3");   // 26-35
    put(36, "C1");   // 36-45
    put(46, "C2");   // 46-55
    put(56, "C3");   // 56-65
    put(66, "B1");   // 66-72
    put(73, "B2");   // 73-79
    put(80, "B3");   // 80-85
    put(86, "A1");   // 86-90
    put(91, "A2");   // 91-94
    put(95, "A3");   // 95-97
    put(98, "S");    // 98-100
}};

public String getLevelByScore(int score) {
    return LEVEL_MAP.floorEntry(score).getValue();
}
```

### 4.3 雷达图 6 维计算

| 维度 | 计算方式 |
|---|---|
| 攻击 | 杀球、网前推挑相关题得分加权 |
| 防守 | 接杀、接吊、双打防守题得分 |
| 网前 | 搓放、勾对角、推球题 |
| 底线 | 高远、平高、底线吊球题 |
| 步伐 | 自评 + 移动相关题 |
| 控球 | 落点准确性、变化能力相关题 |

每维归一化到 0-100，绘制雷达图。

## 5. 战绩 ELO 积分算法（v2）

> v1 不上线，先预留数据结构。

### 5.1 经典 ELO 公式

```
新积分 = 旧积分 + K × (实际得分 - 期望得分)

期望得分 E_a = 1 / (1 + 10^((R_b - R_a) / 400))
实际得分 = 1（胜）或 0（负）
K 值：新手 32，老手 16
```

### 5.2 双打 ELO 处理

```
队伍 ELO = (队员1 ELO + 队员2 ELO) / 2
赛后双方加减积分相同（队员1 和队员2 各分一半）
```

### 5.3 段位 ↔ ELO 映射

```
ELO 1000  → D1
ELO 1100  → D2
ELO 1200  → D3
ELO 1300  → C1
...
ELO 2200+ → S
```

每 100 ELO 一个段位（A 级以上密度更大）。

## 6. 战报海报数据生成

### 6.1 MVP 评选规则

```java
public Member selectMVP(Activity activity) {
    return activity.members.stream()
        .map(m -> new Score(
            m,
            wins(m) * 3.0           // 胜场权重
            + avgPointDiff(m) * 1.0 // 平均分差
            + topScore(m) * 0.5     // 单场最高分
        ))
        .max(Comparator.comparing(Score::value))
        .map(Score::member)
        .orElse(null);
}
```

### 6.2 最佳搭档

胜率最高的两人组合（至少组队 2 次以上）。

### 6.3 黑马

水平最低但胜率最高的人（D-C 级且胜率 > 60%）。

## 8. 附近活动排序算法（v1.1 新增）

### 8.1 问题定义

**输入**：
- 用户经纬度 `(userLng, userLat)`、所属行政区 `regionCode`
- 用户筛选条件（距离、时间段、模式、缺人数、段位）
- 候选活动集合：状态 = `RECRUITING`、`is_public = 1`、开始时间在未来 7 天内

**输出**：
- 按综合权重降序排列的活动列表

### 8.2 综合权重公式

```
score = w1 × distanceScore
      + w2 × timeUrgencyScore
      + w3 × shortageScore
      + w4 × organizerCreditScore
      + w5 × levelMatchScore
      - penalty_blacklisted
```

默认权重（可在若依字典里配置后热加载）：

| 权重 | 默认值 | 含义 |
|---|---|---|
| `w1` 距离 | 0.35 | 越近越高 |
| `w2` 时间临近度 | 0.25 | 越临近开始时间越高（开始前 2-6h 最高） |
| `w3` 缺人紧迫度 | 0.20 | 缺得越多越高，但临近开始时反而希望少缺 |
| `w4` 召集人信誉 | 0.15 | 召集人办过的活动数、好评率 |
| `w5` 段位匹配 | 0.05 | 用户段位在活动 limit_levels 范围内才有分 |

### 8.3 各子分计算

#### 8.3.1 距离分

```java
// 简单球面距离（Haversine 公式）
double distanceKm = haversine(userLat, userLng, venueLat, venueLng);

// 距离分（0-100）
double distanceScore;
if (distanceKm <= 1.0)      distanceScore = 100;
else if (distanceKm <= 3.0) distanceScore = 80 - (distanceKm - 1) * 10;
else if (distanceKm <= 5.0) distanceScore = 60 - (distanceKm - 3) * 10;
else if (distanceKm <= 10.0)distanceScore = 40 - (distanceKm - 5) * 6;
else                        distanceScore = 0;
```

#### 8.3.2 时间临近度分

```java
long hoursUntilStart = Duration.between(now, activity.startTime).toHours();
double timeScore;
if (hoursUntilStart < 0)        timeScore = 0;       // 已开始
else if (hoursUntilStart <= 6)  timeScore = 100;    // 今晚最热
else if (hoursUntilStart <= 24) timeScore = 80;     // 明天
else if (hoursUntilStart <= 72) timeScore = 60;     // 3 天内
else if (hoursUntilStart <= 168)timeScore = 40;     // 一周内
else                            timeScore = 20;
```

#### 8.3.3 缺人紧迫度分

```java
int shortage = activity.plannedCount - activity.currentCount;
int strangerCap = activity.limitStrangersCount - activity.acceptedStrangersCount;
int realShortage = Math.min(shortage, strangerCap);  // 真能让陌生人加入的数量

double shortageScore;
if (realShortage <= 0)      shortageScore = 0;        // 已满，不在列表
else if (realShortage == 1) shortageScore = 60;
else if (realShortage == 2) shortageScore = 80;
else                        shortageScore = 90;

// 越临近开始，缺人紧迫度越高
if (hoursUntilStart < 6 && realShortage > 0) {
    shortageScore = Math.min(100, shortageScore + 20);
}
```

#### 8.3.4 召集人信誉分

```java
User organizer = userRepository.findById(activity.creatorId);
int organized = organizer.totalOrganized;
int credit = organizer.creditScore;

double creditNorm = Math.min(100, credit);  // 100 上限
double organizedBonus;
if (organized < 3)      organizedBonus = 0;
else if (organized < 10) organizedBonus = 10;
else if (organized < 30) organizedBonus = 20;
else                     organizedBonus = 30;

double organizerScore = creditNorm * 0.7 + organizedBonus * 0.3;
```

### 8.4 候选集生成（Step 1）

```sql
SELECT * FROM t_activity
WHERE is_public = 1
  AND status = 'RECRUITING'
  AND accept_strangers = 1
  AND start_time > NOW()
  AND start_time < DATE_ADD(NOW(), INTERVAL 7 DAY)
  AND current_count < planned_count
  AND accepted_strangers_count < limit_strangers_count
  AND ST_Distance_Sphere(
        POINT(venue_lng, venue_lat),
        POINT(:userLng, :userLat)
      ) / 1000 <= :maxRadiusKm
ORDER BY start_time ASC
LIMIT 200;  -- 候选限制
```

> `ST_Distance_Sphere` 是 MySQL 8 内置函数，配合 spatial 索引可用。
> 候选限制 200 条防止排序爆炸。

### 8.5 排序与分页（Step 2）

候选集在内存中按上述权重排序，分页返回 20 条/页。

### 8.6 缓存策略

- Key: `discovery:nearby:{regionCode}:{filterHash}`
- TTL: 5 分钟
- 失效触发：活动状态变化（满员、取消、新增公开活动）通过事件总线主动失效

### 8.7 性能要求

- 单次查询 ≤ 500ms（含 DB + 排序 + 序列化）
- 候选 200 条以下时排序 ≤ 50ms
- 并发 100 QPS 不影响其他接口

### 8.8 边界场景

| 场景 | 处理 |
|---|---|
| 用户没授权位置 | 用 region_code 查同区活动，距离分用区中心点估算 |
| 候选 < 5 条 | 自动放宽半径（×2），最多扩到 50km |
| 候选 = 0 | 返回"附近 50km 内暂无活动"+ 推荐"创建活动" |
| 用户段位为空 | 段位匹配分给 50 中等分，不直接过滤 |

### 8.9 测试用例

```
TC1: 用户在 5km 内有 10 个活动，按距离主排序
TC2: 用户位置造假到南京但活动在上海 — 不应展示
TC3: 召集人黑名单中的用户 — 不应看到该召集人活动
TC4: 段位 D1 用户看不到限 A-S 段位活动
TC5: 活动开始前 1 小时 + 缺人 1 — 排第一
TC6: 同距离同时间，召集人信誉高的优先
TC7: 用户拒绝位置授权 — 用 region 兜底
```

## 9. 信用分变化算法（v1.1 新增）

### 9.1 触发事件与分值

| 事件 | 分值 | 说明 |
|---|---|---|
| 准时到场 | +1 | 活动结束后自动判定 |
| 放鸽子（未到场）| -20 | 首次仅警告不扣分 |
| 临时退出（开始前 ≥ 1h）| 0 | 不扣分 |
| 临时退出（开始前 < 1h）| -10 | 影响召集人凑人 |
| 被召集人投诉 | -30 | 需运营审核通过 |
| 被举报有效 | -50 | 严重违规 |
| 收到召集人感谢 | +5 | 选择性奖励 |
| 月度活跃奖励 | +5 | 月内参加 ≥ 4 场活动 |

### 9.2 自动恢复机制

- 信用分每月 1 号自动恢复，恢复速度 = (100 - 当前分) × 0.1
- 例：当前 60 分，月底自动 +4 分
- 上限 100 分（活跃奖励可达 150）

### 9.3 信用分等级

| 分数 | 等级 | 影响 |
|---|---|---|
| 100-150 | 信誉极佳 | 申请加入自动获得"优先审核" |
| 80-99 | 信誉良好 | 正常 |
| 60-79 | 一般 | 申请加入时显示"警示标签"给召集人 |
| < 60 | 灰名单 | 不能申请陌生人活动，可参加熟人活动 |
| 持续 < 30 一个月 | 临时封禁 | 自动通知运营审核 |

### 9.4 公平性保护

- 用户可申诉，运营 24h 内处理
- 召集人滥用"投诉"扣信用 → 召集人本身信用扣 20
- 同一召集人对同一用户 30 天内只能投诉 1 次

## 10. 训练计划生成算法（v1.2 新增）

### 10.1 问题定义

**输入**：
- 用户当前段位 `currentLevel`（如 C2）
- 目标段位 `targetLevel`（默认 currentLevel + 1）
- 评测雷达图 6 维分数 `radarScores`（攻击/防守/网前/底线/步伐/控球）
- 训练频率 `weeklyFrequency`（默认每周 2 次）
- 每次时长 `weeklyDurationMin`（默认 60 分钟）
- 场地条件 `venueType`（HOME / COURT_NO_PARTNER / COURT_WITH_PARTNER）
- 计划周期 `totalWeeks`（4 / 8 / 12 周）

**输出**：
- 完整训练计划 JSON：每周主题 + 每日任务 + 推荐菜单/视频

### 10.2 算法主流程

```java
public TrainingPlan generate(TrainingPlanRequest req) {
    // Step 1: 找出用户的"弱项技能"
    List<Skill> weakSkills = findWeakSkills(req.user, req.radarScores, req.targetLevel);

    // Step 2: 根据目标段位筛选可练技能（避免推超出能力的技能）
    List<Skill> candidates = filterByLevel(weakSkills, req.currentLevel, req.targetLevel);

    // Step 3: 按周分配，前期打基础后期组合
    List<WeekPlan> weeks = scheduleWeeks(candidates, req.totalWeeks, req.weeklyFrequency);

    // Step 4: 为每个任务选训练菜单和视频
    enrichTasksWithMenuAndVideo(weeks, req.venueType);

    return new TrainingPlan(weeks, calculateStats(weeks));
}
```

### 10.3 弱项识别算法

```java
private List<Skill> findWeakSkills(User user, RadarScores scores, String targetLevel) {
    // 1. 找雷达图最低的 3 个维度
    List<String> weakDims = scores.entrySet().stream()
        .sorted(Map.Entry.comparingByValue())
        .limit(3)
        .map(Map.Entry::getKey)
        .toList();

    // 2. 查询每个维度对应的技能
    List<Skill> result = new ArrayList<>();
    for (String dim : weakDims) {
        List<Skill> skillsInDim = skillRepository.findByCategory(dim);
        // 优先选"用户掌握度低 + 目标段位需要"的技能
        skillsInDim.stream()
            .filter(s -> compareLevel(s.minLevel, targetLevel) <= 0)
            .sorted(Comparator.comparing(s -> getUserMastery(user.id, s.id)))
            .limit(3)
            .forEach(result::add);
    }
    return result;
}
```

### 10.4 周计划排程规则

| 周次 | 主题 | 任务结构 |
|---|---|---|
| 第 1 周 | 基础打牢 | 选 2 个基础技能，每次练 1 个动作 |
| 第 2 周 | 核心进攻/防守 | 选 2-3 个核心技能，加强反复练 |
| 第 3 周 | 组合应用 | 把前 2 周技能组合（如杀球+网前推扑）|
| 第 4 周 | 实战+复测 | 模拟实战练习 + 触发段位复测 |

8 周/12 周计划：把 4 周模板拉伸，前期增加更多基础周，后期增加更多实战周。

### 10.5 训练菜单选择算法

```java
private TrainingMenu pickMenu(Skill skill, String venueType, int availableMin) {
    return menuRepository
        .findBySkillAndVenue(skill.id, venueType)
        .stream()
        .filter(m -> m.durationMin <= availableMin)
        .max(Comparator.comparing(m -> m.qualityScore))
        .orElseGet(() -> getDefaultMenu(skill));  // 兜底菜单
}
```

### 10.6 视频推荐算法

```java
private List<ExternalVideo> pickVideos(Skill skill, String targetLevel, int count) {
    return videoRepository
        .findBySkillAndLevel(skill.id, targetLevel)
        .stream()
        .filter(v -> v.isAvailable)  // 过滤失效视频
        .sorted(Comparator
            .comparing((ExternalVideo v) -> v.usefulCount).reversed()
            .thenComparing(v -> v.qualityScore, Comparator.reverseOrder()))
        .limit(count)
        .toList();
}
```

### 10.9 边界场景

| 场景 | 处理 |
|---|---|
| 用户没有评测记录 | 引导先评测，否则生成"通用 4 周计划" |
| 评测雷达图各项都很差 | 全部排基础类技能 |
| 评测雷达图各项都很好 | 排战术 + 心理对抗类高阶技能 |
| 没有匹配菜单（家里训练但只有陪练菜单）| 用兜底"独自练习"菜单 |
| 视频外链全失效 | 显示"暂无推荐视频" + 文字训练菜单 |
| 用户中途改训练频率 | 重新生成剩余周计划 |
